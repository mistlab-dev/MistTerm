## 【P0】一、SSH Config 导入

### 1.1 背景

现有用户从其他工具迁移至 MistTerm 时，需要手动创建几十个会话。几乎所有竞品都支持从 `~/.ssh/config` 一键导入，这是迁移的第一需求。

### 1.2 功能描述

解析标准 OpenSSH `~/.ssh/config` 文件，将每个 `Host` 配置块转换为 MistTerm `SessionConfig`，批量导入。

### 1.3 解析规则

#### 1.3.1 支持的指令

| ssh_config 指令 | 映射到 SessionConfig | 说明 |
|----------------|---------------------|------|
| `Host` | 自动生成名称 | 取第一个值作为会话名称 |
| `HostName` | `host` | 必填，无此字段跳过该条目 |
| `Port` | `port` | 默认 22 |
| `User` | `username` | 默认为空 |
| `IdentityFile` | 关联 CredentialVault | 记录路径，连接时使用 |
| `ProxyJump` | 新增 `proxy_type: Jump` | 跳板机模式 |
| `ProxyCommand` | 新增 `proxy_type: Command` | 自定义代理命令 |

#### 1.3.2 不支持的指令（忽略不报错）

- `HostKeyAlias` / `CheckHostIP` / `StrictHostKeyChecking`（安全策略由底层 SSH 库处理）
- `LocalForward` / `RemoteForward`（端口转发后续 P1 再做）
- `ServerAliveInterval` / `ServerAliveCountMax`（后续 P1 再做）

#### 1.3.3 Host 命名策略

```
Host 指令值        → 会话名称
  *                 → 跳过（通配符不导入）
  example.com       → "example.com"
  web-*             → 跳过
  prod-db server1   → "prod-db"（取第一个值）
```

如果生成的名称与已有会话重名，在名称后追加 ` (2)`, ` (3)` 等递增后缀。

### 1.4 触发器

**方式一：自动检测**
- 应用启动时检测 `~/.ssh/config` 是否存在未被导入的 Host
- 若发现新待导入条目 > 0，在左侧栏顶部显示：「检测到 N 个未导入的 SSH 配置」+「导入」按钮
- 用户点击后弹窗确认

**方式二：手动操作**
- 菜单「文件 → 导入 SSH 配置」
- 或底部状态栏右键菜单

### 1.5 UI 弹窗

```
┌───────── SSH Config 导入 ─────────┐
│                                     │
│  找到以下 SSH 配置，共 8 个：         │
│                                     │
│  ☑ prod-db      → 192.168.1.10:22  │
│  ☑ prod-web     → 192.168.1.11:22  │
│  ☑ staging-api  → stage.my.com:22  │
│  ☑ dev-server   → 10.0.0.5:2200   │
│  ☐ bastion      → (HostName 缺失)  │
│                                     │
│   [取消]    [导入所选 (3)]           │
└─────────────────────────────────────┘
```

- 每个条目左侧显示 `☐` 勾选框
- HostName 缺失的行置灰、不可选，并标注原因
- 底部显示选中数量
- 确认后批量创建会话

### 1.6 数据边界

| 场景 | 行为 |
|------|------|
| `~/.ssh/config` 不存在 | 不显示导入提示，菜单项置灰 |
| 文件格式错误（非标准 ssh_config） | 尝试逐行解析，成功行导入，错误行在弹窗中列出 |
| 所有条目都已导入过 | 提示「所有 SSH 配置已导入」 |
| 通配符 Host | 跳过，不在弹窗中列出 |
| 导入后原文件被修改 | 重新检测时对比已导入条目的 `Host + HostName` 组合，只显示新增的 |
| 会话数量超过 100 条 | 弹窗中分页显示，每页 20 条 |

### 1.7 实现文件清单

| 文件 | 改动 |
|------|------|
| `src/core/ssh_config_importer.rs` | **新建** — 解析逻辑 |
| `src/ui/ssh_config_import_dialog.rs` | **新建** — 导入弹窗 UI |
| `src/ui/app.rs` | 增加导入触发逻辑和状态字段 |
| `src/core/session.rs` | `SessionConfig` 增加 `ssh_config_marker: Option<String>` 标记已导入的 Host 值 |

**解析建议**：sshdoc crate 没有直接解析器，建议基于 `ssh_config::SshConfig` crate（`ssh-config` 或手写简单行解析器只处理支持的指令）。

## 【P0】二、命令执行历史搜索（Ctrl+R）

### 2.1 背景

当前 `FragmentManager` 已记录每条命令的 `usage_count` / `last_used_at` / `last_success`。但缺少一个按时间倒序展示"最近执行过的命令"的快捷面板。用户想重复执行一条不久前用过的命令时，只能手动重新输入。

### 2.2 功能描述

按 Ctrl+R 打开一个搜索覆盖层（类似 bash reverse-i-search），展示最近执行过的命令列表，支持实时模糊搜索和快速插入。

### 2.3 数据结构

```rust
pub struct CommandHistory {
    /// 历史记录（内存中最多 500 条）
    pub entries: Vec<HistoryEntry>,
    /// 持久化文件路径
    persist_path: PathBuf,
}

pub struct HistoryEntry {
    /// 命令文本
    pub command: String,
    /// 执行时间戳
    pub executed_at: chrono::DateTime<chrono::Local>,
    /// 来源会话 ID（允许回溯到哪个服务器执行的）
    pub session_id: Option<String>,
    /// 来源会话名称
    pub session_name: Option<String>,
    /// 执行结果：true=成功, false=失败
    pub success: bool,
}
```

### 2.4 存储与生命周期

| 项目 | 值 |
|------|-----|
| 内存上限 | 500 条（超过时淘汰最旧的） |
| 持久化 | `~/.config/mistterm/command_history.json` |
| 文件上限 | 1000 条 |
| 生命周期 | 自动清理 60 天前的记录 |
| 多会话合并 | 所有会话的历史合并到一个列表中 |
| 启动加载 | 异步读文件（不阻塞 UI） |

### 2.5 UI 覆盖层

```
Ctrl+R 在当前终端窗口内展开一个覆盖层（overlay）：
┌─ 搜索命令历史 (Ctrl+R 继续) ────────────────────────┐
│ ❯ docker                                               │
│                                                        │
│  → docker ps -a                        # myserver      │
│    docker logs -f app                 # prod-web       │
│    docker compose up -d               # dev-server     │
│    docker images --format "{{.ID}}"    # myserver      │
│                                                        │
│  共 4 条结果                               [Esc 关闭]  │
└────────────────────────────────────────────────────────┘
```

- **输入搜索词**：实时过滤，大小写不敏感
- **上下箭头**：选择条目
- **Enter**：插入选中命令到终端（替换当前输入行）
- **继续按 Ctrl+R**：跳到下一条匹配（循环查找）
- **Esc**：关闭覆盖层，不修改输入
- 选中条目高亮背景，箭头图标指示

### 2.6 与现有片段系统的关系

| 维度 | 片段库 (Fragment) | 命令历史 (CommandHistory) |
|------|-------------------|--------------------------|
| 内容来源 | 用户手动创建/导入 | 自动记录 |
| 数据特点 | 精心维护、分类管理 | 原始日志、无分类 |
| 使用场景 | 常用命令模板 | 临时回查最近操作的命令 |
| 去重 | 否（用户自主管理） | 是（连续重复只保留最新一条） |
| 搜索 | 全局搜索 | Ctrl+R 即时搜索 |

两者互补，不合并。

### 2.7 数据边界

| 场景 | 行为 |
|------|------|
| 无终端连接 | Ctrl+R 不响应或提示「请先连接」 |
| 命令过长 (>1000 字符) | 截断到 500 字符展示，实际命令完整保留 |
| 敏感命令（含密码） | 不自动过滤——用户可以手动从历史中删除 |
| 历史记录达到上限 | 淘汰最旧的，持久化时自动清理 |
| 连续相同命令 | 只保留最新一条（更新 executed_at） |
| 文件损坏 | 清空历史，不弹错误 |

### 2.8 删除历史

- 在覆盖层 hover 条目时显示「🗑」按钮
- 点击即删除单条
- 无二次确认（可撤销场景：仅从内存删除，保存时才真正写入）

### 2.9 实现文件清单

| 文件 | 改动 |
|------|------|
| `src/core/command_history.rs` | **新建** — 数据结构 + 持久化 + 搜索 |
| `src/ui/command_history_overlay.rs` | **新建** — Ctrl+R 覆盖层渲染 |
| `src/ui/terminal.rs` | 按键分发：Ctrl+R 调起覆盖层，插入选中命令 |
| `src/ui/app.rs` | 集成 CommandHistory 实例（MistTermApp 新增字段） |

## 【P0】三、终端输出日志自动保存

### 3.1 背景

用户排查问题时经常需要回顾刚才的终端输出。当前屏幕滚动有限，断开连接后输出全部丢失。所有主流竞品都支持会话日志保存。

### 3.2 功能描述

每个 SSH 连接的终端输出自动追加写入日志文件，连接断开后日志保留，支持按会话/日期浏览。

### 3.3 存储结构

```
~/.config/mistterm/logs/
├── {session_id_1}/
│   ├── 2026-05-17.log      // 当天所有输出的纯文本
│   └── 2026-05-16.log
├── {session_id_2}/
│   └── 2026-05-17.log
└── ...
```

### 3.4 日志格式

```
[MistTerm 2026-05-17 10:30:15] ← 连接建立
[MistTerm 2026-05-17 10:30:15] user@192.168.1.10:22
❯ ls -la
total 64
drwxr-xr-x  8 root root 4096 May 17 10:00 .
drwxr-xr-x  3 root root 4096 May 15 09:00 .
❯ docker ps -a
CONTAINER ID   IMAGE     COMMAND   CREATED    STATUS    PORTS     NAMES
[MistTerm 2026-05-17 11:02:10] ← 连接断开
```

- 每个命令提示符前写入时间戳标记
- 连接建立/断开写入系统事件标记
- 纯文本格式（不含 ANSI 转义码），可用 `cat` / `less` / `grep` 直接查看
- 每行以 `❯` 开头表示用户输入的原始命令

### 3.5 配置项

在偏好设置 > 终端 > 增加：

| 配置项 | 类型 | 默认值 | 说明 |
|--------|------|--------|------|
| 自动保存日志 | bool | true | 开关 |
| 日志保存天数 | number | 30 | 超过此天数的日志自动清理 |
| 日志目录 | path | `~/.config/mistterm/logs/` | 可自定义 |
| 包含 ANSI 颜色 | bool | false | 保存原始 ANSI 还是净文本 |
| 日志文件大小上限 | number | 50MB | 单文件超过后切割为 `.1.log`, `.2.log` |

### 3.6 日志浏览

- 在左侧会话列表右键菜单增加「查看日志」
- 打开一个只读 `Window` 展示日志内容（egui 的 `TextEdit` 只读模式）
- 顶部有搜索框和时间范围筛选

### 3.7 性能要求

| 场景 | 要求 |
|------|------|
| 高频输出（tail -f 级别） | 异步 IO 写入，不阻塞终端渲染 |
| 大文件（100MB+） | 日志浏览时只读取末尾 500KB + 搜索范围 |
| 批量清理 | 启动时后台线程执行，不阻塞 UI |

### 3.8 实现伪代码

```rust
struct SessionLogger {
    file: Option<BufWriter<File>>,
    buffer: Vec<u8>,
    last_flush: Instant,
}

impl SessionLogger {
    /// 新连接的日志初始化
    fn start_log(session_id: &str) -> Result<()>;
    /// 追加输出（不立即 flush，由定时器或连接断开时 flush）
    fn write_output(&mut self, data: &[u8]);
    /// 写入命令提示符时间戳
    fn write_prompt_marker(&mut self);
    /// 连接关闭时 flush 并关闭文件
    fn stop_log(&mut self);
    /// 清理过期日志
    fn cleanup_old_logs(retention_days: u32);
}
```

### 3.9 数据边界

| 场景 | 行为 |
|------|------|
| 写入权限不足 | 静默关闭该会话的日志，不弹错误 |
| 磁盘空间不足 (< 100MB) | 停止写入并跳过旧日志清理（保留现有） |
| 会话从未连接过 | 不创建日志目录 |
| 日志目录手动删除 | 下次写入时自动重建 |
| 单会话单日输出巨大 (>500MB) | 在 50MB 时自动切割文件 |

### 3.10 实现文件清单

| 文件 | 改动 |
|------|------|
| `src/core/session_logger.rs` | **新建** — 日志写入 + 清理 |
| `src/core/session_manager.rs` | 在 `connect`/`disconnect` 中调用 `SessionLogger` |
| `src/ui/terminal.rs` | 输出渲染时同时传递给 `SessionLogger` |
| `src/ui/settings_dialog.rs` | 增加日志配置项 UI |
| `src/core/config.rs` | 增加日志配置字段 |

## 【P0】四、KeepAlive / ServerAliveInterval 配置

### 4.1 背景

SSH 连接可能因防火墙/NAT 超时被断开。用户需要在不重启应用的情况下保持连接活跃。当前代码已有 `ssh_auto_reconnect`（自动重连），但缺少主动保活机制。

### 4.2 功能描述

在 SSH 会话中设置保活心跳包（SSH 协议级 `keepalive`），检测连接存活状态，断开后触发自动重连。

### 4.3 实现方式

- SSH 连接建立后，底层 SSH 库设置 `ServerAliveInterval` 和 `ServerAliveCountMax`
- 使用 SSH 协议 `SSH_MSG_IGNORE` 作为心跳（与 OpenSSH 的 ServerAliveInterval 一致）
- 不需要额外 TCP 包

### 4.4 配置项

**在会话编辑面板中**：

| 配置项 | 类型 | 默认值 | 说明 |
|--------|------|--------|------|
| 启用心跳保持 | bool | true | 连接级开关 |
| 心跳间隔（秒） | number | 30 | 范围 5-300 |
| 心跳超时次数 | number | 3 | 连续 N 次无回应判定断开 |
| 断开后自动重连 | bool | true | 复用现有 `ssh_auto_reconnect` |

**全局默认值在偏好设置 > 连接**：

| 配置项 | 类型 | 默认值 |
|--------|------|--------|
| 心跳间隔 (全局默认) | number | 30 |
| 心跳超时次数 (全局默认) | number | 3 |
| 断开后自动重连 (全局默认) | bool | true |

### 4.5 与自动重连的关系

```
心跳丢失 → 判定断开 → 自动重连（如果开启）
                                       ↑
                       等待 retry_interval → 重试次数 < max_retries
                                               ↓ 是
                                          重新连接
                                               ↓ 否
                                          停止重试，标记为离线
```

### 4.6 会话编辑面板 UI 变化

在会话编辑对话框中的"高级"区域增加：

```
┌─ 连接保活 ──────────────────────────┐
│ ☑ 启用心跳保持                       │
│   间隔: [30  ] 秒   超时次数: [3]    │
│ ☑ 断开后自动重连                     │
└──────────────────────────────────────┘
```

### 4.7 数据边界

| 场景 | 行为 |
|------|------|
| 连接已断开 | 不再发送心跳 |
| 心跳间隔设为 0 | 等价于关闭保活 |
| 配置修改 | 下次连接时生效，已建立连接不热重载 |
| 全局默认值修改 | 不影响已有连接的单独设置 |

### 4.8 实现文件清单

| 文件 | 改动 |
|------|------|
| `src/ssh/connection.rs` | 在 SSH 连接参数中设置 ServerAliveInterval |
| `src/core/session.rs` | `SessionConfig` 增加 `keepalive_enabled`, `keepalive_interval`, `keepalive_count_max`, `keepalive_auto_reconnect` |
| `src/ui/dialogs.rs` (会话编辑) | 增加 KeepAlive 配置 UI |
| `src/core/config.rs` | 全局默认保活配置 |

## 【P0】五、会话排序 & 颜色标签

### 5.1 背景

当前左侧会话列表按创建顺序排列，没有排序选项。用户在管理 20+ 台服务器时，无法快速定位"最常用的"或"最近连过的"。颜色标签可以辅助视觉区分开发/测试/生产环境。

### 5.2 功能描述

- 支持按 4 种排序方式排列会话列表
- 支持给会话设置颜色标签（红/黄/绿/蓝/紫/灰）
- 颜色标签在会话名左侧显示为 6px 实心圆点

### 5.3 排序方式

放在侧栏搜索框下方的分类标签区：

```
[全部] [在线] [离线] │ [▼ 排序: 最近连接]
```

点击排序按钮弹出：

```
┌─ 排序方式 ──┐
│ ○ 名称 (A→Z)│
│ ○ 名称 (Z→A)│
│ ● 最近连接  │ ← 默认
│ ○ 创建时间  │
└──────────────┘
```

| 排序方式 | 规则 |
|---------|------|
| 名称 (A→Z) | 按 `name` 自然排序（字符串升序） |
| 名称 (Z→A) | 按 `name` 自然排序（字符串降序） |
| 最近连接 | 按 `last_connected_at` 降序，从未连接的排在最后 |
| 创建时间 | 按 `created_at` 降序（新创建的在上方） |

### 5.4 颜色标签

在会话编辑面板增加颜色选择器：

```
标签颜色: [○ 无] [● 红] [● 黄] [● 绿] [● 蓝] [● 紫] [● 灰]
```

侧栏渲染：

```
  ┌──── 会话列表 ──────┐
  │ ●  prod-db        │  ← 绿点
  │ ●  prod-web       │  ← 绿点
  │ ●  staging-api    │  ← 蓝点
  │ ○  dev-server     │  ← 无标签（灰色描边）
  └─────────────────────┘
```

颜色映射：

| 标签 | 16进制色值 | 推荐场景 |
|------|-----------|---------|
| 红 | `#f44336` | 生产环境/紧急 |
| 黄 | `#ffc832` | 预发布/警告 |
| 绿 | `#4CAF50` | 正常/线上稳定 |
| 蓝 | `#2196F3` | 开发环境 |
| 紫 | `#9C27B0` | 测试/预发布 |
| 灰 | `#9E9E9E` | 下线/暂停使用 |

### 5.5 数据边界

| 场景 | 行为 |
|------|------|
| 排序改变 | 仅改变展示顺序，不改变持久化存储顺序 |
| 标签不设置 | 默认灰色描边小圆点（与未选中 Tab 一致） |
| 标签批量修改 | 不提供批量操作，用户需逐个编辑 |
| 过滤+排序同时作用 | 先过滤（全部/在线/离线），再排序 |

### 5.6 实现文件清单

| 文件 | 改动 |

| `src/ui/dialogs.rs` (会话编辑) | 增加颜色选择器 |

## 【P1】六、Tab 分屏

### 6.1 背景

当前 MistTerm 的每个 Tab 是一个完整终端窗口。运维场景经常需要同时看日志和监控（或两台服务器比对），切 Tab 太慢。FinalShell / Tabby / MobaXterm 标配分屏。

### 6.2 功能描述

每个 Tab 内部支持分割为多个终端列（左右分屏或上下分屏），每个子列可连接不同的服务器，或同一服务器的不同 shell 进程。

### 6.3 分屏模型

```rust
pub enum SplitDirection {
    Horizontal, // 左右分
    Vertical,   // 上下分

pub struct SplitPane {
    pub direction: SplitDirection,
    pub ratio: f32,              // 0.0~1.0，分割线位置
    pub left_terminal:  Option<SplitTerminal>,
    pub right_terminal: Option<SplitTerminal>,
    // 或递归嵌套：
    pub left_split:  Option<Box<SplitPane>>,
    pub right_split: Option<Box<SplitPane>>,
}

pub struct SplitTerminal {
    pub session_id: String,
    pub rendered: bool,
}
```

**支持递归嵌套**：左右分屏的每一侧可以再进行上下分屏，以此类推。

### 6.4 操作方式

| 操作 | 方式 |
|------|------|
| 左右分屏 | 右键 Tab 标签 →「左右分屏」 |
| 上下分屏 | 右键 Tab 标签 →「上下分屏」 |
| 切换分屏焦点 | Alt+方向键 (←→↑↓) |
| 调整分割比例 | 拖动分割线（光标变 ↔ 或 ↕） |
| 合并分屏 | 拖动子标签到父标签合并 |
| 关闭分屏 | 子标签的关闭按钮（X） |
| 限制 | 最大分屏数：一个 Tab 内最多 4 个子窗口 |

### 6.5 分割线 UI

```
┌──────────────────────┬─────────────────┐

│                      │                  │
│                      │                  │
├──────────────────────┴─────────────────┤
│          状态栏                         │
└─────────────────────────────────────────┘
```

- 分割线宽度：2px，颜色：`border` 主题色
- hover 时高亮，拖动时实时跟随
- 分割线上显示一个可点击的「合并」按钮（中间小圆点 hover 显示）

### 6.6 数据边界

| 场景 | 行为 |
|------|------|

| 分屏中一个连接断开 | 该子屏显示"已断开"，其他不受影响 |

| 拖动分割线到极端位置 | 最小宽度 200px，达不到自动折叠为 Tab |

## 【P1】七、跳板机 / 代理链

### 7.1 背景

企业网络架构中，生产服务器通常不能直连，需要通过跳板机（Bastion）中转。当前 MistTerm 只支持直连，用户需要手动执行 `ssh -J user@bastion host` 或者先用 `ssh -N -L` 建立隧道。

### 7.2 功能描述

在会话配置中支持：
- **ProxyJump**：通过跳板机转发 SSH 连接（`ssh -J`）
- **ProxyCommand**：通过任意命令建立代理隧道（`nc -X connect ...`）
- **多级跳板**：`ssh -J bastion1,bastion2 target` 形式

### 7.3 数据模型

```rust
pub enum ProxyType {
    /// 直连（无代理）
    Direct,
    /// ProxyJump（SSH 内置跳板）
    Jump {
        /// 跳板机连接串，格式同 ssh -J 参数
        /// 支持多级跳板："bastion1,bastion2"
        proxy_hosts: Vec<JumpHost>,
    },
    /// ProxyCommand（自定义命令）
    Command {
        /// 代理命令模板，例如：ssh -W %h:%p bastion
        proxy_command: String,
    },
}

pub struct JumpHost {
    pub username: String,
    pub host: String,
    pub port: u16,
    pub identity_file: Option<String>,
}
```

### 7.4 会话编辑 UI

在会话编辑对话框的"高级"区域增加代理设置：

```
[○] 直连  [●] 跳板机  [○] 代理命令

跳板机配置：
┌────────────────────────────────────────┐
│  host: user@bastion.example.com:22     │
│  ┌────────────────────────────────────┐│
│  │ user@bastion1.example.com:22    ✕  ││
│  │ user@bastion2.example.com      ✕  ││
│  └────────────────────────────────────┘│
│  [+ 添加跳板]                           │
├────────────────────────────────────────┤
│  SSH 密钥 (可选): [选择...]              │
└────────────────────────────────────────┘
```

### 7.5 连接流程

```
用户点击连接 → 检查 ProxyType
                ├─ Direct → 直接建立 SSH 连接
                ├─ Jump   → 检查缓存：跳板连接是否已建立
                │             ├─ 已建立 → 通过复用通道直连目标
                │             └─ 未建立 → 建立到跳板机的连接
                │                        ├─ 成功 → 通过跳板机建立目标连接
                │                        └─ 失败 → 回退到直连（可选）
                └─ Command → 启动代理命令进程
                              ├─ 成功 → SSH 连接到 stdio
                              └─ 失败 → 显示错误
```

### 7.6 数据边界

| 场景 | 行为 |
|------|------|
| 跳板机不可达 | 显示"无法连接跳板机：{跳板机地址} {失败原因}" |
| 多级跳板中间节点不可达 | 显示具体哪个节点失败 |
| 跳板机已断开 | 所有依赖该跳板的连接断开并自动重连 |
| 跳板机和目标使用相同凭据 | 支持跳板机密钥独立指定 |
| SSH config 已配置 ProxyJump | 导入时自动识别 |

## 【P1】八、批量多机执行

### 8.1 背景

运维场景中经常需要同时在多台服务器执行相同命令（如查看进程、更新配置），当前只能逐台连接操作。FinalShell / MobaXterm / SecureCRT / electerm 都支持批量执行。

### 8.2 功能描述

用户选择多个会话连接，在底部弹出批量执行面板，输入一行命令后并发发送到所有选定连接的终端。

### 8.3 UI 交互

#### 8.3.1 触发

在左侧会话列表勾选多个连接：

```
  │ ○ dev-server       │    │ [发送到所选]            │
  └─────────────────────┘    └─────────────────────────┘
```

**操作路径**：
1. 侧栏每个会话左侧增加勾选框（点击时切换选中）
2. 勾选 ≥2 个后底部面板展开
3. 输入命令，点击发送或按 Ctrl+Enter
4. 命令并发发送，输出在面板中逐台展示

#### 8.3.2 结果展示

```

│ ┌──── staging-api ──── ✓ 0.5s ───────────────────┐     │
```

### 8.5 技术选型

| 项目 | 方案 |

| 命令为空 | 发送按钮置灰 |

### 8.7 数据边界

| 场景 | 行为 |

## 【P2】十、AI 命令助手

### 10.1 背景

Warp 的核心卖点是 AI 辅助——自然语言描述需求，自动生成 shell 命令。electerm 也集成了 AI 生成命令。这是 Rust 原生 TUI 之外的差异化竞争力。

### 10.2 功能描述

在终端输入框上方或右侧提供一个 AI 输入条，用户输入自然语言（中文/英文），调用 LLM API 生成对应的 shell 命令并自动插入终端。

### 10.3 交互流程

```
1. 用户在终端输入区按 Ctrl+I 或点击 AI 按钮
2. 输入描述（如"查找最近修改的5个文件"）
3. AI 返回建议命令：
   ┌─ AI 建议 ─────────────────────────┐
   │ find . -type f -printf '%T@ %p\n'  │
   │   | sort -rn | head -5             │
   │ [插入] [解释] [重新生成]           │
   └────────────────────────────────────┘
4. 用户点击「插入」→ 命令填入终端输入区
5. 用户点击「解释」→ AI 分步解释该命令的作用
6. 用户点击「重新生成」→ 再次调用 LLM 获取新版本
```

### 10.4 LLM 配置

在偏好设置 > AI > 增加：

| 配置项 | 类型 | 默认值 | 说明 |
|--------|------|--------|------|
| 启用 AI | bool | false | 开关，默认关闭 |
| API 地址 | string | `http://127.0.0.1:11434/v1` | Ollama 或兼容 OpenAI 的 API |
| 模型名 | string | `qwen2.5:7b` | 建议先设为本地可运行的模型 |
| API Key | string | 空 | 非本地模型需要 |
| 超时时间 | number | 30 | 秒 |
| 系统提示词 | string | `你是一名 Linux 运维专家，请将自然语言描述转换为精确的 shell 命令。只返回命令本身，不要额外解释。` | 可自定义 |

### 10.5 命令解释

用户选中一条命令后按 `Ctrl+Shift+I`，AI 对命令进行分步解释：

```
┌─ AI 解释 ───────────────────────────┐
│ find . -type f -printf '%T@ %p\n'    │
│ │ sort -rn | head -5                 │
│                                       │
│ 1. find . -type f                     │
│    → 递归查找当前目录下所有普通文件   │
│ 2. -printf '%T@ %p\n'                │
│    → 输出格式：修改时间戳+文件名      │
│ 3. sort -rn                           │
│    → 按时间戳降序排列（最新在最前）   │
│ 4. head -5                            │
│    → 只取前5行（最新的5个文件）       │
└───────────────────────────────────────┘
```

### 10.6 错误分析

当命令执行返回非零退出码时，自动在终端下方显示一个 AI 提示图标，点击后 AI 分析错误输出并提供修复建议。

```
# 用户执行：

┌─ [🤖 分析错误] ───────────────────────────┐

```

## 【P2】十一、网络诊断（Ping / Traceroute）

### 11.1 背景

### 11.2 功能描述

### 11.3 诊断面板

```

│    icmp_seq=2 ttl=64 time=0.6ms  │
│    icmp_seq=3 ttl=64 time=0.4ms  │
│    icmp_seq=4 ttl=64 time=0.7ms  │
│                                   │
│  ── 统计 ──                       │
│  包: 4/4 ✓  延迟: min/avg/max    │
```

### 11.4 实现方式

```rust
pub enum DiagnosticType {
    Ping { count: u32, interval_ms: u32 },
    Traceroute { max_hops: u32 },
```

## 【P2】十二、拖拽上传

### 12.1 背景

用户从本地 Finder / 资源管理器选择文件上传到服务器，当前需要通过 SFTP 面板手动导航目录再上传。MobaXterm / Tabby / FinalShell 支持直接拖拽终端窗口上传。

### 12.2 功能描述

从操作系统文件管理器将文件或文件夹拖拽到 MistTerm 的终端区域，自动执行 SCP/SSH 上传到当前连接的远程服务器的当前工作目录。

### 12.3 交互流程

```
1. 选择本地文件/文件夹
2. 拖拽到终端区域（出现蓝色半透明 overlay 指示）
3. 释放鼠标 → 弹出确认弹窗
   ┌─ 上传文件 ──────────────────────┐
   │ 文件: report.pdf (2.3 MB)        │
   │ 上传到: prod-db:/var/log/        │
   │ 目标目录: /var/log/app/          │
   │  [传输] [取消]                    │
   └───────────────────────────────────┘
4. 点击传输 → SFTP 上传
5. 进度显示在状态栏或 SFTP 面板：
   ┌─ report.pdf  ▓▓▓▓▓▓▓░░░  75% ──┐
   │ 速度: 1.2 MB/s   剩余: 2s      │
   └─────────────────────────────────┘
```

### 12.4 目标目录确定规则

```
优先级：
1. 当前终端 shell 的工作目录（通过 `pwd` 或 SSH shell integration 获取）
2. 回退：该会话的默认远程目录（`SessionConfig.remote_dir`）
3. 再回退：`/root/` 或 `/home/{username}/`
```

### 12.5 实现方案

```rust
pub struct DragDropUpload {
    /// SFTP 通道（复用已有连接）
    pub sftp: Option<SftpChannel>,
    /// 上传队列
    pub queue: Vec<UploadJob>,
}

pub struct UploadJob {
    pub local_path: PathBuf,
    pub remote_dir: String,
    pub is_folder: bool,
    pub progress: f32,       // 0.0 ~ 1.0
    pub speed_bytes_ps: u64,
}
```

**技术要点**：
- egui 内拖拽检测：使用 `egui::DND` 或 `raw_input.dropped_files`
- 文件路径通过 `cc.egui_ctx.input().raw.dropped_files` 获取
- SFTP 上传复用已有连接（不走 SCP）

### 12.6 数据边界

| 场景 | 行为 |
|------|------|
| 无活跃连接 | 拖入时不显示 overlay，释放后无反应 |
| 目标目录不存在 | 提示"目录不存在"并提供 [创建目录] 按钮 |
| 同名文件存在 | 提示"文件已存在，覆盖？" |
| 上传大文件 (> 500MB) | 提示"大文件建议使用 SFTP 面板异步传输" |
| 文件夹上传 | 递归上传，保持目录结构 |
| 多个文件 | 队列上传，显示总体进度 |
| 连接在上传中中断 | 上传失败，显示错误，支持手动重试 |

### 12.7 实现文件清单

| 文件 | 改动 |
|------|------|
| `src/core/drag_drop_upload.rs` | **新建** — 拖拽处理 + SFTP 上传 |
| `src/ui/terminal.rs` | 接收 `dropped_files` 事件，显示 overlay 和进度 |
| `src/ui/app.rs` | 集成 DragDropUpload 实例 |

## 【P2】十三、远程文件编辑

### 13.1 背景

用户有时需要在远程服务器上快速编辑配置文件（如 nginx.conf, .env）。当前需要下载 → 本地编辑 → 上传。FinalShell / SecureCRT / MobaXterm 内置了远程文件编辑器。

### 13.2 功能描述

在 SFTP 面板中双击文件，通过系统默认编辑器或内置简易编辑器打开远程文件。保存时自动回传到服务器。

### 13.3 交互流程

```
1. SFTP 面板中双击文件（如 /etc/nginx/nginx.conf）
2. 临时下载到本地临时目录
3. 用系统默认文本编辑器打开，或打开内置简易编辑器
4. 用户编辑并保存
5. 文件变更检测：保存后自动重新上传到原路径
```

### 13.4 编辑器选择

| 优先级 | 检测条件 | 编辑器 |
|--------|---------|--------|
| macOS | `$EDITOR` 环境变量 | 用户自定义 |
| macOS | `open` 命令 | 系统关联编辑器 (TextEdit / VSCode) |
| Linux | `$EDITOR` 环境变量 | 用户自定义 |
| Linux | `xdg-open` | 系统关联编辑器 |
| 兜底 | — | 内置简易编辑器（基于 egui TextEdit，纯文本）|

### 13.5 内置简易编辑器

```
┌─ 编辑: /etc/nginx/nginx.conf (prod-db) ─── [Ctrl+S 保存] ─┐
│                                                              │
│ user  nginx;                                                 │
│ worker_processes  auto;                                      │
│                                                              │
│ error_log  /var/log/nginx/error.log warn;                    │
│ pid        /var/run/nginx.pid;                               │
│                                                              │
│ events {                                                     │
│     worker_connections  1024;                                │
│ }                                                             │
│                                                              │
│  Ln 1, Col 1    修改未保存                                   │
└──────────────────────────────────────────────────────────────┘
```

- 支持基本语法高亮（通过简易关键词匹配）
- 行号显示
- Ctrl+S 保存并上传
- 标题栏显示文件路径 + 服务器名 + 修改状态

### 13.6 上传策略

| 事件 | 行为 |
|------|------|
| Ctrl+S | 立即上传，不等待 |
| 关闭编辑器 | 如果内容已修改，弹出「保存更改？[保存] [不保存] [取消]」 |
| 上传失败 | 显示错误，文件留在本地临时目录，不丢失修改内容 |
| 远程文件被其他人修改 | 上传前检查 `mtime`，冲突时提示「远程文件已被修改，覆盖？」 |

### 13.7 数据边界

| 场景 | 行为 |
|------|------|
| 文件 > 5MB | 提示"文件过大，建议在服务器上直接用 vim 编辑" |
| 二进制文件 | 检测 MIME 类型，二进制文件禁用编辑 |
| 无写权限 | 下载后提示"文件只读，上传可能失败" |
| 临时目录清理 | 应用退出时自动清理 |
| 多文件同时编辑 | 每个文件一个独立窗口 |

### 13.8 实现文件清单

| 文件 | 改动 |
|------|------|
| `src/core/remote_file_editor.rs` | **新建** — 下载/上传/变更检测 |
| `src/ui/remote_editor_window.rs` | **新建** — 内置编辑器窗口 |
| `src/ui/sftp_panel.rs` | 文件列表双击事件触发编辑 |
