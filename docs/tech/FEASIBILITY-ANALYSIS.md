| 评级 | 说明 |
|-----|------|

| ❌ **不可行** | 当前技术栈无法实现或成本过高 |

### 1.3 SFTP 文件传输

| 功能 | 可行性 | 技术难度 | 说明 |
|-----|-------|---------|------|
| SFTP 协议 | ⚠️ 部分可行 | ⭐⭐⭐⭐ | libssh2 支持有限 |
| 文件浏览 | ⚠️ 部分可行 | ⭐⭐⭐ | 需要实现 SFTP 客户端 |
| 上传/下载 | ⚠️ 部分可行 | ⭐⭐⭐⭐ | 需要完整 SFTP 实现 |
| 拖拽传输 | ⚠️ 部分可行 | ⭐⭐⭐ | egui 拖拽支持有限 |
| 断点续传 | ⚠️ 部分可行 | ⭐⭐⭐⭐ | 需要 SFTP 协议支持 |
| **lrzsz 协议** | ✅ **可行** | ⭐⭐⭐ | **必须实现，一期核心** |

**lrzsz 优先级调整**:
- ⚠️ **从"二期"提升到"一期必须"**
- 原因：运维高频场景，终端内直接传输
- 技术：ZMODEM 协议公开，实现相对简单

**SFTP vs lrzsz**:
| 维度 | SFTP | lrzsz |
|-----|------|-------|
| 协议复杂度 | 高（SSH 子协议） | 中（ZMODEM） |
| 实现难度 | ⭐⭐⭐⭐ | ⭐⭐⭐ |
| 使用频率 | 中 | **高** |
| 用户体验 | 需要切换窗口 | **终端内直接传输** |
| 优先级 | P1 | **P0** |

**建议**:
- ✅ **一期必须完成 lrzsz**
- ⚠️ SFTP 可以二期或砍掉（lrzsz 已覆盖大部分场景）

## 三、lrzsz 协议详细分析

### 3.1 为什么 lrzsz 是必须的？

| 场景 | 频率 | 痛点 |
|-----|------|------|
| 上传配置文件 | 每天多次 | 需要切换 SFTP 工具 |
| 下载日志文件 | 每天多次 | 需要切换 SFTP 工具 |
| 批量传输脚本 | 每周多次 | 需要切换 SFTP 工具 |
| 快速分享小文件 | 每天 | 需要其他工具 |

**核心优势**:
```
传统方式:
用户 → SSH 终端 → 输入 rz → 切换 SFTP 工具 → 选择文件 → 上传

lrzsz 方式:
用户 → SSH 终端 → 输入 rz → 自动弹出文件选择 → 上传完成
```

### 3.3 技术实现方案

**核心组件**:

```rust
// 1. 终端输出检测
struct LrzszDetector {
    buffer: String,
    state: LrzszState,
}

impl LrzszDetector {
    fn detect(&mut self, output: &str) -> Option<LrzszEvent> {
        // 检测 "rz ready" 或 "sz filename"
        if output.contains("rz ready") {
            Some(LrzszEvent::UploadReady)
        } else if let Some(filename) = parse_sz_command(output) {
            Some(LrzszEvent::DownloadReady(filename))
        } else {
            None
        }
    }
}

// 2. ZMODEM 协议实现
struct ZmodemTransfer {
    state: ZmodemState,
    file_info: Option<FileInfo>,
    data_buffer: Vec<u8>,
    crc: u32,
}

impl ZmodemTransfer {
    async fn upload(&mut self, file_path: &str, channel: &mut SshChannel) -> Result<()> {
        // 发送 ZRQINIT
        self.send_zrqinit(channel).await?;
        
        // 发送文件头
        self.send_file_header(file_path, channel).await?;
        
        // 发送数据块
        self.send_data_blocks(file_path, channel).await?;
        
        // 发送结束
        self.send_eof(channel).await?;
        Ok(())
    }
    
    async fn download(&mut self, file_path: &str, channel: &mut SshChannel) -> Result<()> {
        // 接收文件头
        self.recv_file_header(channel).await?;
        
        // 接收数据块
        self.recv_data_blocks(channel).await?;
        
        // 保存文件
        self.save_file(file_path).await?;
        Ok(())
    }
}

// 3. 文件选择器集成
fn show_file_picker(is_upload: bool) -> Option<PathBuf> {
    #[cfg(target_os = "macos")]
    {
        use rfd::FileDialog;

    }
}
```

**状态机**:

```rust
enum LrzszState {
    Idle,              // 空闲
    Detecting,         // 检测命令
    UploadReady,       // 等待上传文件
    DownloadReady,     // 等待下载路径
    Transferring,      // 传输中
    Completed,         // 完成
    Error(String),     // 错误
}

enum ZmodemState {
    WaitingHeader,     // 等待文件头
    SendingHeader,     // 发送文件头
    SendingData,       // 发送数据
    ReceivingData,     // 接收数据
    Verifying,         // 验证 CRC
    Finished,          // 完成
}
```

## 六、总结

### 6.1 核心结论

1. **当前技术栈可行**: Rust + egui + ssh2 可实现核心 SSH 终端功能
2. **高级功能需后端**: 团队协作、云端同步等需要完整的后端服务
3. **SFTP 是难点**: libssh2 的 SFTP 支持不完善，需要额外工作
4. **建议 MVP 先行**: 先实现核心功能，再迭代高级功能

### 6.2 跨平台支持要求

**必须支持平台**:
- ✅ macOS (10.15+)
- ✅ Windows (10+)
- ✅ Linux (Ubuntu 18.04+, Fedora 33+)

**影响的功能**:
| 功能 | 跨平台方案 | 工作量 |
|-----|----------|-------|
| 云端同步 | WebDAV + 第三方云 | +15 人天 |
| 系统密钥链 | keyring crate | +5 人天 |
| 终端渲染 | 统一库 | 已完成 |
| SSH 连接 | ssh2 crate | 已完成 |

### 6.2 推荐路线

```
Phase 1 (2-3 个月): MVP
├── SSH 连接（已完成 70%）
├── 单标签终端（已完成 50%）
├── 会话管理（已完成）

└── 基础主题定制

Phase 2 (3-4 个月): 增强

```
